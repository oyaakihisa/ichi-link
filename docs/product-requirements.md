# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称

**ichi-link** - 現場向け位置情報変換ツール兼マップランチャー

### プロダクトコンセプト

- **地図中心の操作体験**: 地図を常時表示し、検索・変換・確認・共有をすべて地図上で完結させる
- **安全な変換**: 入力タイプを明示的に指定することで、曖昧な自動推定を避け確実な変換を提供
- **即座に地図起動**: 変換結果から複数の地図サービス（Google Maps、Yahoo!地図、Apple Maps、地理院地図）を即座に開ける
- **現場設備の確認**: 公開設備情報（AED、消火栓など）を地図上に重ねて表示し、周辺設備の確認を支援する

### プロダクトビジョン

緊急対応の現場で受け取った位置情報を「迷わず使える形」に変える受付ハブを提供する。
住所・緯度経度・表記ゆれのある位置情報を短時間で安全に正規化し、複数の地図サービスで即座に開ける状態にする。
測地系や形式の違いによる誤認を減らし、緊急時に「変換のための考える時間」をゼロにする。

このアプリは「変換ツールを開いてから地図を見る」のではなく、**「地図の中で位置を確定・確認・共有する」**アプリである。
さらに、位置を確定した後、**その周辺のAEDや消火栓をすぐに確認できる**ことで、現場対応を支援する。

### 目的

- 位置情報の形式や測地系の違いを吸収し、現場で即利用できる形に変換する
- 複数の地図サービスへのアクセスを一元化し、地図起動までの手間を削減する
- 曖昧な入力に対して適切な警告を出し、誤変換による現場事故を防ぐ
- GISの専門知識がなくても安全に位置情報を扱えるようにする
- **公開データを使った位置確認レイヤーとして、周辺のAEDや消火栓を地図上で確認できるようにする**

### 公開設備レイヤーの前提

- 今回追加する設備レイヤーは「運用管理システム」ではなく、**公開データに基づく位置確認レイヤー**として位置づける
- 消火栓については、「位置情報」と「実際の使用可否」は別概念である。MVPでは位置情報レイヤーとして扱い、利用可否の保証は行わない
- 将来的に防火水槽、医療機関、避難場所などの公開設備も同様の構造で追加可能な設計とする

## 成功指標(KPI)

### プライマリーKPI

- **変換成功率**: 主要入力パターンの95%以上を正しく処理できる（MVP時点）
- **地図起動までの時間**: 入力から地図起動まで5秒以内（ネットワーク遅延除く）
- **ユーザビリティスコア**: 初見ユーザーが3分以内に基本操作を完了できる

### セカンダリーKPI

- **警告表示の適切さ**: 曖昧な入力に対して100%警告を表示する（false negative ゼロ）
- **継続利用率**: MVP導入後1ヶ月で週1回以上利用するユーザーが80%以上
- **現場評価**: 現場ヒアリングで「実用になる」と評価される

### 運用品質指標（改善判断用）

- **入力タイプ別成功率**: 住所 / WGS84 / Tokyo Datum それぞれの変換成功率
- **警告発生率**: 全変換に対する警告表示の割合
- **外部地図起動率**: パネル表示後に外部地図を開いた割合
- **コピー/共有利用率**: 各コピーボタン・共有機能の利用頻度

### POIレイヤー関連指標

- **AED / 消火栓レイヤー表示成功率**: レイヤー有効化時のPOI表示成功率
- **POI詳細表示時間**: 地図表示後、対象POIをタップして詳細パネル表示するまでの時間
- **現場評価（POI）**: 現場ヒアリングで「周辺設備確認に使える」と評価される
- **アプリ内POI確認率**: 外部地図起動前にアプリ内でPOI確認が完了する割合

## 機能要件

### コア機能(MVP)

#### 1. 位置情報の入力受付

**ユーザーストーリー**:
通信指令担当者として、受け取った位置情報をすぐに処理するために、地図上で素早く検索できる入力機能が欲しい

**受け入れ条件**:

- [ ] 地図上部にフローティング検索バーを表示する
- [ ] 入力タイプをタブ形式で切り替える（常時展開、3つのタブが並ぶ）
  - [ ] 住所（国土地理院 地名検索APIによるジオコーディング対応）
  - [ ] 世界測地系（WGS84）座標
  - [ ] 旧日本測地系（Tokyo Datum）座標
- [ ] 各タブに適切なプレースホルダーを表示する（例: 「東京都千代田区丸の内1-1-1」「35.6812, 139.7671」）
- [ ] 緯度経度（十進度形式）を受け付ける（例: 35.6812, 139.7671）
- [ ] 緯度経度（度分秒形式）を受け付ける（例: 35°40'52"N 139°46'2"E）
- [ ] カンマ区切り、空白区切り、記号付き表記に対応する
- [ ] コピー&ペーストで入力できる
- [ ] IME入力中のEnterキー誤送信を防止する

**入力揺れ耐性**:

以下の表記ゆれに対応し、正しく解釈する:

- 前後の空白（トリム処理）
- 全角数字・記号の混在
- 緯度経度の逆順入力（警告付きで処理）
- 度分秒の記号ゆれ（° ′ ″ / ° ' " / 度 分 秒）
- N/E/S/W付き表記（例: 35°40'52"N）
- 判定不能な入力に対しては安全に失敗し、エラーメッセージを表示する

**優先度**: P0(必須)

#### 2. 入力内容の形式判定

**ユーザーストーリー**:
通信指令担当者として、座標の形式を意識せずに使うために、システムが十進度か度分秒かを自動判定してほしい

**受け入れ条件**:

- [ ] 座標入力時、十進度か度分秒かを自動判定する
- [ ] 判定結果をスライドパネル上部に明示する（例: 「WGS84座標として判定」）
- [ ] 判定に失敗した場合、エラーメッセージを表示する
- [ ] 入力タイプ（住所/WGS84/Tokyo）はユーザーがタブで指定するため、測地系の推定は不要

**優先度**: P0(必須)

#### 3. 測地系の変換

**ユーザーストーリー**:
通信指令担当者として、測地系の違いによる誤差（約400-500m）を防ぐために、WGS84と旧日本測地系間の確実な変換が欲しい

**受け入れ条件**:

- [ ] WGS84（世界測地系）座標の入力・出力をサポートする
- [ ] 旧日本測地系（Tokyo Datum）座標の入力・出力をサポートする
- [ ] WGS84 → Tokyo Datum への変換に対応する
- [ ] Tokyo Datum → WGS84 への変換に対応する
- [ ] 入力タイプごとに測地系が確定しているため、推定や手動切り替えは不要

**優先度**: P0(必須)

#### 4. 出力の標準化

**ユーザーストーリー**:
通信指令担当者として、変換結果を確認・共有するために、標準化された形式で出力してほしい

**受け入れ条件**:

- [ ] WGS84形式の緯度経度を表示する
- [ ] 旧日本測地系（Tokyo Datum）形式の緯度経度を表示する
- [ ] 小数点以下6桁の精度で表示する（約10cm精度）
- [ ] 各出力値を個別にコピーできる

**優先度**: P0(必須)

#### 5. 地図URL生成・起動

**ユーザーストーリー**:
通信指令担当者として、変換結果をすぐに地図で確認するために、複数の地図サービスで開けるリンクが欲しい

**受け入れ条件**:

- [ ] Google MapsのURLを生成し、ワンクリックで開ける
- [ ] Yahoo!地図のURLを生成し、ワンクリックで開ける
- [ ] Apple MapsのURLを生成し、ワンクリックで開ける
- [ ] 地理院地図のURLを生成し、ワンクリックで開ける
- [ ] 各URLを個別にコピーできる

**優先度**: P0(必須)

#### 6. 安全確認UI（警告表示）

**ユーザーストーリー**:
通信指令担当者として、誤変換による現場事故を防ぐために、曖昧な入力に対して警告を確認したい

**目的**:
警告表示は「単に注意を促す」ためではなく、**誤変換による現場事故を防ぐための確認導線**である。
ユーザーが警告を確認したうえで外部地図を開く流れを確保する。

**受け入れ条件**:

- [ ] 入力タイプ、変換後の座標、地図リンクを一覧表示する
- [ ] 緯度経度の順番が不自然な場合に警告を表示する（例: 経度が先に入力された可能性）
- [ ] 日本国内想定だが海外座標の可能性がある場合に警告を表示する
- [ ] 住所入力時、入力文字列と実際に採用した地名・住所が異なる場合に情報を表示する
  - 例: 「石川県輪島市町野町広江3-2-1」と入力 →「石川県輪島市町野町広江」までの情報で位置を特定した場合、その旨を明示
- [ ] 警告は赤色などの目立つ色で表示し、ユーザーが見落とさないようにする
- [ ] 警告がある場合でも地図起動は可能とするが、ユーザーが警告を視認できる導線にする

**優先度**: P0(必須)

#### 7. コピー・共有機能

**ユーザーストーリー**:
通信指令担当者として、変換結果を他のシステムや同僚に共有するために、簡単にコピー・共有できる機能が欲しい

**受け入れ条件**:

- [ ] 各出力値（座標、住所、URL）ごとにコピーボタンを配置する
- [ ] 「全部コピー」ボタンで主要情報をまとめてコピーできる
- [ ] コピー成功時に視覚的フィードバックを表示する（ボタンの変化など）

**全部コピーの対象項目**:

1. 住所（存在する場合）
2. 世界測地系（WGS84）座標
3. 旧日本測地系（Tokyo Datum）座標
4. Google Maps URL

**共有機能**:

- [ ] LINE共有に対応する
  - モバイル: LINE Share URL を使用
  - PC: LINE Social Plugins を使用
- [ ] Web Share API に対応する（対応ブラウザのみ）
- [ ] 共有テキストには以下を含める:
  - 世界測地系（WGS84）座標
  - 旧日本測地系（Tokyo Datum）座標
  - Google Maps URL

**優先度**: P0(必須)

#### 8. マップ表示機能

**ユーザーストーリー**:
出動中の消防士として、現在地周辺や指定場所を地図で確認するために、正確なマップ表示が欲しい

**受け入れ条件**:

- [ ] Mapbox（Zenrinデータ使用）によるマップを表示する
- [ ] **マップをアプリのメイン画面として常時表示する**
- [ ] 日本の建物情報が正確に表示される
- [ ] 地図ラベルを日本語で表示する
- [ ] 初期表示は日本全体を中心とする
- [ ] ピンチ操作で拡大縮小できる
- [ ] ドラッグ操作で移動できる
- [ ] ナビゲーションコントロール（ズームボタン）を表示する
- [ ] 地図スタイルを切り替えられる（標準地図 / 衛星画像）
  - [ ] 右上にスタイル切り替えボタンを配置する
  - [ ] 衛星画像はラベル付き（satellite-streets-v12）を使用する
  - [ ] スタイル切り替え後も日本語ラベル設定を維持する

**地図カメラ挙動ルール**:

- [ ] 変換成功時は対象地点へスムーズにズーム移動する（flyTo）
- [ ] ズーム移動時、ピンがスライドパネルに隠れない位置（画面上部1/3付近）に表示する
- [ ] ズームレベルは地点確認に十分なレベル（ズームレベル16程度）まで寄る
- [ ] 長押し時はカメラ移動を行わない（ユーザーが意図した位置を維持）

**優先度**: P0(必須)

#### 9. ピン表示・状態管理

**ユーザーストーリー**:
通信指令担当者として、検索結果や長押し位置を地図上で確認するために、ピンによる位置表示が欲しい

**ピン状態管理ルール**:

地図上の選択状態を常に1つに保ち、現場での混乱を防ぐため、以下のルールを適用する。

- [ ] **アクティブピン**（検索/変換/長押しで生成）は常に1件のみとする
- [ ] 新しい検索結果が来た場合、既存のアクティブピンを新しいピンで置き換える
- [ ] 新しい長押しが来た場合、既存のアクティブピンを新しいピンで置き換える
- [ ] 検索結果と長押しは相互に排他的（一方を実行すると他方のピン・パネルをクリア）
- [ ] スライドパネルを閉じてもピンは地図上に残す
- [ ] **現在地マーカー**は別レイヤーとして扱い、アクティブピンの置換対象に含めない（将来実装時）
- [ ] **POIピン**（AED、消火栓など）は背景レイヤーとして複数表示できる
- [ ] **アクティブピンとPOI選択は排他的**:
  - POIをタップするとアクティブピンをクリアし、POI詳細パネルを表示
  - 検索/長押しを実行するとPOI選択をクリアし、アクティブピンを配置

**受け入れ条件**:

- [ ] 変換成功時に該当座標にピンを自動配置する
- [ ] 長押し時に該当座標にピンを配置する
- [ ] ピンは視認しやすい色（青色）で表示する

**優先度**: P0(必須)

#### 10. 長押しピン機能

**ユーザーストーリー**:
通信指令担当者として、地図上の任意の場所の座標を取得するために、長押しでピンを立てる機能が欲しい

**受け入れ条件**:

- [ ] マップ上を長押し（約500ms）するとピンが立つ
- [ ] ピンを立てると下からスライドパネルが表示される
- [ ] 長押し中に指を動かした場合（10px以上）、長押しをキャンセルする
- [ ] ピン位置から逆ジオコーディングで住所を取得する
- [ ] 住所取得中はローディング表示を行う
- [ ] WGS84座標とTokyo Datum座標を計算・表示する

**優先度**: P0(必須)

#### 11. 位置情報スライドパネル

**ユーザーストーリー**:
通信指令担当者として、検索結果やピンを立てた場所の情報を確認・共有するために、詳細パネルが欲しい

**位置づけ**:
スライドパネルは単なる補助UIではなく、**結果確認・共有・外部地図起動の中心UI**である。
検索結果、長押し結果、POI詳細のすべてで同一のパネルコンポーネントを使用し、一貫した操作体験を提供する。

**受け入れ条件**:

- [ ] 下からスライドして表示されるパネル
- [ ] **変換実行時に自動でパネルを表示する**
- [ ] **長押しピン設置時にもパネルを表示する**
- [ ] **POIタップ時にもパネルを表示する**
- [ ] パネル上部に共有ボタン群を配置（LINEで共有、共有、全部コピー）
- [ ] パネルに以下を表示:
  - **入力/取得方法の表示**（検索結果の場合のみ）
    - 「住所として判定」
    - 「WGS84座標として判定」
    - 「Tokyo Datum座標として判定」
    - ※長押し取得の場合は入力タイプ表示なし
  - **警告**（曖昧な入力の場合、検索結果のみ）
  - 住所
  - 世界測地系（WGS84）座標
  - 旧日本測地系（Tokyo Datum）座標
  - **地図で開くリンク**（検索結果の場合: Google Maps、Yahoo!地図、Apple Maps、地理院地図）
  - **Google Mapsリンク**（長押しの場合: Google Mapsのみ）
- [ ] 各座標値・住所を個別にコピーできる
- [ ] パネルを×ボタンで閉じることができる
- [ ] オーバーレイ（背景の半透明黒）をタップしてもパネルを閉じられる
- [ ] パネルの最大高さは画面の70%とし、内容が多い場合はスクロール可能にする

**POI詳細表示時の追加項目**:

パネルは結果の種類に応じて表示項目が変わる。

**AEDタップ時**:
- 名称
- 種別（AED）
- 住所
- 緯度経度（WGS84）
- 設置場所詳細（取得できる場合）
- 利用可能時間（取得できる場合）
- 小児対応有無（取得できる場合）
- 外部地図リンク（Google Maps）
- 共有

**消火栓タップ時**:
- 名称または識別情報
- 種別（消火栓）
- 住所または位置説明
- 緯度経度（WGS84）
- 外部地図リンク（Google Maps）
- 共有
- ※「利用可否は別途確認が必要です」の注記を表示

**POI外部地図リンクについて**:
- MVPでは簡素化のため、POI詳細からの外部地図リンクはGoogle Mapsのみとする
- 将来的に他の地図サービス（Yahoo!地図、Apple Maps等）への拡張余地は残す

**優先度**: P0(必須)

### POIデータ取得設計

#### データ取得戦略

POIデータの取得は、全国全件を初回に一括取得するのではなく、地図の表示範囲に応じて必要なPOIのみを取得する。

**基本方針**:

- 初回に全国POIを全件取得しない
- 地図の表示範囲（ビューポート）に応じて必要なPOIだけ取得する
- 取得条件は `bbox`（表示範囲）、`type`（POI種別）、`zoom`（ズームレベル）を基本とする
- AEDと消火栓はカテゴリ別に扱う
- 低ズームではPOIを絞る、またはクラスタ中心で表示する

**取得タイミング**:

- 地図の移動（pan）またはズーム操作の終了時に`bbox`を取得する
- API呼び出しはデバウンスする（連続操作時の過剰リクエスト防止）
- 同一周辺範囲の取得結果はキャッシュする

**ズームレベル連動**:

- 低ズーム（広域表示）: POI件数を制限、またはクラスタ表示のみ
- 高ズーム（詳細表示）: 範囲内の全POIを表示
- 消火栓は低ズームでは表示抑制しやすい設計とする

#### 自前API前提

フロントエンドは公的APIを直接叩かず、自前APIを経由してPOIデータを取得する。

**理由**:

- 公的ソースのレスポンス遅延、レート制限、返却形式の差異をアプリから切り離す
- 地図表示に最適化したデータ形式で配信できる
- キャッシュ戦略を自由に設計できる
- 将来的なVector Tiles移行への布石

**データフロー**:

```
[公開データソース] → 定期取得 → [自前DB] → [自前API] → [フロントエンド]
```

**想定API**:

```
GET /pois?bbox={west},{south},{east},{north}&types=aed,fireHydrant&zoom={zoom}
GET /pois/{id}
```

- `bbox`: 表示範囲（west,south,east,north）
- `types`: 取得するPOI種別（カンマ区切り）
- `zoom`: 現在のズームレベル（返却件数や詳細度の調整に使用）

**レスポンス設計方針**:

- 一覧取得（`GET /pois`）では、パネル初回表示に必要な最小限の項目を返す
  - id, type, name, latitude, longitude, address など
- 詳細情報（設置場所詳細、利用可能時間、小児対応有無など）は`GET /pois/{id}`で個別取得
- これにより、一覧取得のレスポンスサイズを抑えつつ、必要に応じて詳細を補完できる
- 将来的にVector Tiles / PMTilesへ移行しやすいよう、クライアントとデータ配信方式を疎結合にする

#### 更新戦略

公開データは定期的に自前DBへ同期する。

**更新頻度**:

- 日次〜週次での同期を基本とする
- リアルタイム更新は不要

**保持情報**:

- データソース（source）
- ソース側の更新日時
- 自前DBへの反映日時

**考慮事項**:

- 更新失敗時にも前回取得データと更新日時が確認できること
- 古いデータの扱いに配慮する（表示時に更新日時を参照可能にする）
- 消火栓の位置情報を最新化しても、利用可否までは保証しない

#### 描画方式

大量のPOIを効率的に描画するため、ReactコンポーネントによるDOMマーカーではなく、Mapbox layer / sourceベースの描画を採用する。

**採用方式**:

- GeoJSONソース + Circleレイヤーによる描画
- Mapbox GL JSのクラスタリング機能を活用
- POI更新時は`source.setData()`で差分更新

**クラスタリング**:

- 低ズームで多数のPOIが重なる場合はクラスタ表示
- クラスタタップで展開またはズームイン
- Mapboxネイティブのcluster機能を使用

#### 将来の拡張性

現段階では過剰設計を避けつつ、以下への移行余地を残す。

**PMTiles / Vector Tiles移行**:

- 想定規模が10万件を超える場合の選択肢
- サーバーレス配信が可能
- クライアント側の描画ロジックは大きく変わらない

**カテゴリ追加**:

- 共通POIデータモデルに従えば、防火水槽、医療機関、避難場所なども同様の構造で追加可能
- レイヤー切替UIは拡張を想定した設計とする

### 共通POIデータモデル

将来の拡張性を担保するため、AED / 消火栓を含むPOIは共通の構造で扱う。

**共通項目**:
- id: 一意識別子
- type: POI種別（aed / fireHydrant / 将来: fireCistern, medicalFacility, evacuationSite など）
- name: 名称
- latitude: 緯度（WGS84）
- longitude: 経度（WGS84）
- address: 住所
- detailText: 設置場所詳細などの補足情報
- availabilityText: 利用可能時間などの可用性情報
- source: データソース
- updatedAt: データ更新日時

**将来拡張候補**:
- 防火水槽（fireCistern）: 次の拡張候補
- 医療機関（medicalFacility）
- 避難場所（evacuationSite）

## 非機能要件

### パフォーマンス

- **入力判定・変換時間**: 入力後500ms以内に判定結果を表示する
- **地図URL生成**: 即時（100ms以内）
- **画面遷移**: 最小限（基本的に1画面で完結）
- **初回ロード時間**: 3秒以内（3G回線を除く）
- **マップ読み込み時間**: 初回3秒以内、以降1秒以内
- **ピン反応時間**: 長押し検出後100ms以内にピン表示
- **逆ジオコーディング**: ピン設置後1秒以内に住所取得
- **地図操作**: 60fps以上のスムーズな操作
- **POI取得**: 初回に全国全件を取得しない
- **POI取得タイミング**: 地図移動/ズーム終了時に必要範囲のみ取得する
- **API呼び出し**: デバウンスにより過剰リクエストを防止する
- **POIキャッシュ**: 同一周辺範囲の取得結果はキャッシュする
- **POI描画**: Mapbox layer/sourceベースで描画し、DOMマーカー大量描画を避ける
- **POIレイヤー表示**: 地図操作を阻害しない範囲でPOIを表示する
- **クラスタリング**: 低ズームでの視認性と描画性能を両立する

### ユーザビリティ

- **習得時間**: 非専門家でも3分以内に基本操作を習得できる
- **入力方式**: 地図上部のフローティング検索バー、タブ式の入力タイプ切替
- **1画面完結**: 地図・検索・結果確認・共有がすべて同一画面内で完結
- **ボタンサイズ**: タッチ操作可能なサイズ（最小44x44px）
- **判定結果**: 分かりやすい日本語で表示する
- **警告表示**: 曖昧な入力には必ず警告を出す
- **レスポンシブ**: スマホでも操作しやすい
- **状態遷移の一貫性**: ピン・パネルの表示/非表示が予測可能な挙動をする
- **操作導線**: 検索→ズーム→パネル表示→確認/共有の流れが自然に繋がる
- **POI視認性**: AEDと消火栓の見た目が一目で区別できる
- **情報過多防止**: 地図上で情報過多になりすぎないよう、レイヤー切替とズーム連動で調整
- **POI詳細導線**: POIタップから詳細確認まで迷わない
- **ズーム連動**: レイヤーON/OFFとズームに応じて自然に情報量が変わる

### 信頼性

- **曖昧性の明示**: 曖昧な変換は明確に曖昧と表示する
- **エラーハンドリング**: エラー時に黙って失敗せず、明確なメッセージを表示する
- **入力値保護**: 入力値の破損や欠落を防ぐ
- **安全な失敗**: 不正な値に対して安全に失敗する
- **誤変換警告**: 誤変換の危険がある場合に警告を表示する
- **POIデータソース明示**: 公開データの更新時刻やソースが確認できる設計が望ましい
- **消火栓利用可否の注記**: 消火栓の利用可否は位置情報とは別概念であることを誤解させない
- **更新失敗時の挙動**: 前回取得データと更新日時が確認できることが望ましい

### セキュリティ

- **データ保存**: 入力された住所・座標を不要に永続保存しない（MVPでは端末ローカルのみ）
- **通信暗号化**: HTTPS必須
- **ログ制限**: ログに生の位置情報を過剰出力しない
- **エラー出力**: エラーや分析用途でも位置情報の扱いに配慮する
- **位置情報の配慮**: 位置情報を扱うアプリとして、プライバシーに最低限の配慮を行う

### スケーラビリティ

- **地図サービス追加**: 新しい地図サービスの追加が容易な設計
- **測地系追加**: 新しい測地系の追加が容易な設計
- **UI調整**: 現場向けにUIを調整しやすい設計
- **将来連携**: 将来的な業務システム連携が可能な設計
- **POIカテゴリ追加**: 今後、他の公開設備レイヤー（防火水槽など）を追加しやすい設計
- **POI UI拡張**: POIカテゴリを増やしてもUIが破綻しにくい設計
- **配信方式移行**: 将来PMTiles/Vector Tilesへ移行しやすいよう、クライアントとデータ配信方式を疎結合にする

## スコープ外

明示的にスコープ外とする項目:

- **全国POI全件の初回一括取得**: ビューポートベース取得を採用
- **ユーザー入力・変換履歴のサーバーサイド永続化**: MVPでは端末ローカル保存のみ（POI公開データ用DBはMVP対象内）
- **ユーザー認証・アカウント管理**: MVPでは不要
- **組織向け管理機能**（利用ログ、操作履歴）: MVPでは不要
- **外部システム連携**（指令システム、業務端末）: MVPでは不要
- **完全なオフライン対応**: MVPでは通信必須
- **音声入力・音声認識**: MVPでは不要
- **多言語対応**: 日本語のみ
- **消火栓のリアルタイム利用可否管理**: MVPでは位置情報レイヤーとして扱う
- **消防内部の非公開設備情報**: 公開データのみを対象
- **設備の保守管理機能**: 運用管理システムではない
- **POI一覧画面 / 高度な検索画面**: 地図中心UIで完結
- **ルート案内**: 外部地図アプリに委譲
- **内部向けメモ機能**: MVPでは不要
- **認証付き運用管理画面**: MVPでは不要
- **PMTiles / Vector Tilesの現時点での本格導入**: 将来の拡張オプションとして残す
